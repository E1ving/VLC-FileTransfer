# VLC-FileTransfer

XMU Computer Networks Course Project – Group 19

## 📖 项目简介

本项目是一套基于 **可见光通信 (Visible Light Communication, VLC)** 的点对点文件传输原型系统。通过显示器发送编码色块序列，利用摄像头捕捉视频并实时解码，实现了复杂物理环境下的高带宽、零误码文件恢复。

项目攻克了 **帧撕裂 (Frame Tearing)**、**采样频率失配 (Nyquist Mismatch)** 以及 **位对齐 (Bit Alignment)** 等 VLC 核心技术难点。

***

## 🚀 核心技术指标 (Optimized)

在标准室内光照环境下测得：

- **有效传输速率 (Goodput):** \~240.96 kbps (优化后)
- **误码率 (BER):** **0.000000%** (位级完全一致，理想条件下)
- **丢失率 (Erasure Rate):** 0.03% (万分之三，理想条件下)
- **同步机制:** 基于 8-bit 序号循环与 Epoch 自动回绕处理
- **校验算法:** 16-bit CRC (循环冗余校验)

***

## 🛠️ 核心架构逻辑

### 1. 物理层采样策略 (Nyquist Sampling)

系统采用 **“高频过采样”** 方案：

- **发送端 (TX):** 稳定 25 FPS 刷新率 (优化后)。
- **接收端 (RX):** 采用 60 FPS 录制。

根据奈奎斯特定理，60Hz 采样能完美覆盖 25Hz 的信号变化。系统通过 **“首胜制 (First-Success-Wins)”** 逻辑，在每一组冗余帧中自动提取第一个校验通过的完整帧，屏蔽掉由于屏幕翻页导致的“撕裂帧”。

### 2. 链路层协议 (Link Layer)

- **帧结构:** `[Header | Seq (1B) | Data Payload (1211B) | CRC16 (2B)]` (优化后)。
- **序号补全:** 若发生物理丢帧，系统会根据序号差值自动在内存中填充 `std_capacity` 长度的 `0` 占位，防止后续数据发生全局位移。
- **自动对齐:** 解码器具备起始序号感知能力。若录制从第 N 帧开始，评估程序会自动跳过原始文件的前 N(N-1) 帧数据进行比对。

### 3. 视觉处理逻辑

- **VisionCorrector:** 利用 OpenCV 的四点透视变换 (`getPerspectiveTransform`) 修正拍摄角度，将倾斜的屏幕画面还原为标准矩形。
- **二值化引擎:** 动态阈值处理，确保在不同亮度的显示器上都能精准识别色块比特位。

### 4. 优化策略

- **BLOCK_SIZE:** 从 12px 优化为 10px，增加数据密度
- **视频分辨率:** 从 1008x1008 优化为 1000x1000，提高数据容量
- **帧率:** 从 15fps 优化为 25fps，提高传输速率
- **编码器:** 使用 MP4V 编码器，提高视频兼容性

## 📂 目录结构说明

### 核心引擎

- **`core/encoder_engine.py`**: 发送端核心。负责将原始二进制文件切片、添加序号头及 CRC 校验码，生成待显示的像素矩阵。
- **`core/decoder_engine.py`**: 接收端核心。负责从校正后的图像中提取比特流，进行校验识别及序号回绕处理。
- **`core/protocol.py`**: 编解码协议定义，包括物理参数和帧结构。

### 工具与脚本

- **`encoder.py`**: 发送端主程序。驱动显示器循环播放编码后的色块序列。
- **`decoder.py`**: **接收端主程序**。集成解帧、比特重组、空洞补全、文件输出及自动化 BER 评估。
- **`utils/VisionCorrector.py`**: 视觉预处理。负责屏幕定位与透视校正（Perspective Transform）。
- **`utils/video_muxer.py`**: 视频处理工具。负责高效的视频/图像序列转换。
- **`bin_genertor.py`**: 二进制文件生成器，用于生成测试数据。

### 数据文件

- **`data/test.bin`**: 测试二进制文件。
- **`data/test_optimized.bin`**: 优化后的测试二进制文件。
- **`in.bin`**: 原始输入文件（真值数据，用于评估比较）。
- **`out.bin`**: 解码后恢复的二进制文件。
- **`vout.bin`**: **传输掩码文件**。记录每一字节的有效性（0xFF 表示该字节来自物理接收，0x00 表示该字节为丢失补全）。

## 🚦 操作指南

### 1. 环境准备

确保安装了 Python 3.10+ 及必要的科学计算库：

```bash
pip install opencv-python numpy
```

### 2. 编码流程

1. **生成测试数据**:
   ```bash
   python bin_genertor.py
   ```

2. **运行编码程序**:
   ```bash
   python encoder.py data/test_optimized.bin output_optimized.mp4 10000
   ```
   - `data/test_optimized.bin`: 输入二进制文件
   - `output_optimized.mp4`: 输出视频文件
   - `10000`: 最大时长限制（毫秒）

### 3. 解码流程

1. **录制视频**:
   - 使用手机以 1080p 60fps 录制生成的视频
   - 确保四个角落的定位点清晰可见
   - 保持手机稳定，避免抖动和倾斜

2. **运行解码程序**:
   ```bash
   python decoder.py input.mp4 output.bin vout.bin
   ```
   - `input.mp4`: 录制的视频文件
   - `output.bin`: 解码后恢复的二进制文件
   - `vout.bin`: 传输掩码文件

### 4. 报告解读

运行结束后，控制台将输出如下报告：

```
========================================
✨ 解码任务圆满完成！
📊 严格评估报告 (对标 in.bin):
   - 有效传输量: 819200 bits
   - 有效传输率: 240.96 kbps
   - 误码率 (BER): 0.000000%
   - 丢失率 (Erasure): 0.03%
========================================
```

## ❓ 常见问题排查 (FAQ)

- **Q: 为什么有效传输量（Effective Bits）非常低？**
  - **A:** 通常是因为起始帧没对齐。请确保发送端和接收端序号一致，或检查 `in.bin` 是否包含多余的 Header。
- **Q: 为什么 Log 中出现大量 CRC 失败？**
  - **A:** 这是 60fps 采样下的正常现象。系统捕捉到了屏幕刷新瞬间的“撕裂图像”，这些无效包已被“首胜制”逻辑安全过滤，不影响最终数据。
- **Q: 录制时有什么要求？**
  - **A:** 建议使用手机的 **AE/AF Lock** 功能锁定曝光和对焦，避免传输中画面亮度闪烁。确保四个角落的定位点清晰可见，保持手机稳定。
- **Q: 视频无法播放怎么办？**
  - **A:** 尝试使用不同的视频播放器，或使用系统默认播放器打开视频。
